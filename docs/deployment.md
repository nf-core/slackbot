# Deployment

The bot uses Slack's **Socket Mode**, which means it maintains an outbound
WebSocket connection to Slack. There's no need for a public URL, load balancer,
or ingress rules. You just need a process running somewhere that stays alive.

## Architecture

```
Slack <──WebSocket──> ECS Fargate (bot) ──> GitHub API
                                        ──> DynamoDB
```

All infrastructure is defined as code using **AWS CDK** (Python) in the `infra/`
directory. Deployments are automated via **GitHub Actions** — pushing to `main`
triggers tests, builds a Docker image, pushes it to GHCR, and deploys to AWS.

## AWS Resources

The CDK stack (`NfCoreBotStack`) creates and manages:

| Resource                 | Details                                                                                                                      |
| ------------------------ | ---------------------------------------------------------------------------------------------------------------------------- |
| **VPC**                  | Public subnets only (2 AZs), no NAT gateway                                                                                  |
| **DynamoDB Table**       | `nf-core-bot`, PK/SK + GSI1, on-demand billing (sites, organisers, registrations only — hackathon metadata is in YAML files) |
| **ECS Cluster**          | `nf-core-bot`                                                                                                                |
| **ECS Fargate Service**  | 1 task, 0.25 vCPU, 512 MB, public IP                                                                                         |
| **CloudWatch Log Group** | `/ecs/nf-core`                                                                                                               |
| **IAM Roles**            | Auto-generated by CDK with least-privilege                                                                                   |

The Docker image is stored in **GitHub Container Registry**
(`ghcr.io/nf-core/nf-core-slackbot`), not in AWS.

Secrets are stored in **SSM Parameter Store** (managed manually, not by CDK).

## Day-to-Day Workflow

**To deploy a change:** push to `main` (or merge a PR). That's it. This includes
hackathon lifecycle changes — editing a YAML file's `status` field and pushing
is how you open, close, or archive a hackathon.

The GitHub Actions workflow (`.github/workflows/deploy.yml`) will:

1. Run lint, type-check, and tests
2. Build the Docker image and push to GHCR
3. Run `cdk deploy` to update infrastructure if needed
4. Force a new ECS deployment to pick up the new image

**To monitor:** check CloudWatch Logs at `/ecs/nf-core`, or the ECS console for
task health. If the bot crashes, ECS restarts it automatically. If a bad
deployment crashes on startup, the circuit breaker rolls back to the previous
working version.

## One-Time Setup

These steps only need to be done once when setting up the bot from scratch.

### 1. Store secrets in SSM Parameter Store

```bash
aws --profile nf-core --region eu-west-1 ssm put-parameter \
  --name /nf-core/SLACK_BOT_TOKEN --type SecureString --value "xoxb-..."
aws --profile nf-core --region eu-west-1 ssm put-parameter \
  --name /nf-core/SLACK_SIGNING_SECRET --type SecureString --value "..."
aws --profile nf-core --region eu-west-1 ssm put-parameter \
  --name /nf-core/SLACK_APP_TOKEN --type SecureString --value "xapp-..."
aws --profile nf-core --region eu-west-1 ssm put-parameter \
  --name /nf-core/GITHUB_TOKEN --type SecureString --value "github_pat_..."
```

### 2. Bootstrap CDK

CDK needs a one-time bootstrap step that creates an S3 bucket and IAM roles it
uses internally:

```bash
pip install aws-cdk-lib
cd infra
cdk bootstrap aws://728131696474/eu-west-1 --profile nf-core
```

### 3. Set up GitHub OIDC for AWS

The deploy workflow uses GitHub's OIDC federation to authenticate with AWS — no
long-lived access keys needed.

#### Create the OIDC identity provider (if it doesn't already exist):

```bash
aws --profile nf-core iam create-open-id-connect-provider \
  --url https://token.actions.githubusercontent.com \
  --client-id-list sts.amazonaws.com \
  --thumbprint-list 6938fd4d98bab03faadb97b34396831e3780aea1
```

#### Create an IAM role for GitHub Actions:

Create a file `github-actions-trust-policy.json`:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Principal": {
        "Federated": "arn:aws:iam::728131696474:oidc-provider/token.actions.githubusercontent.com"
      },
      "Action": "sts:AssumeRoleWithWebIdentity",
      "Condition": {
        "StringEquals": {
          "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
        },
        "StringLike": {
          "token.actions.githubusercontent.com:sub": "repo:nf-core/slackbot:*"
        }
      }
    }
  ]
}
```

```bash
aws --profile nf-core iam create-role \
  --role-name nf-core-bot-github-actions \
  --assume-role-policy-document file://github-actions-trust-policy.json

aws --profile nf-core iam attach-role-policy \
  --role-name nf-core-bot-github-actions \
  --policy-arn arn:aws:iam::aws:policy/AdministratorAccess
```

> **Note:** `AdministratorAccess` is used because CDK needs broad permissions to
> create/manage VPC, ECS, DynamoDB, IAM roles, etc. You can scope this down to a
> custom policy later if desired. The trust policy restricts this role to only
> be assumable by the `nf-core/slackbot` GitHub repository.

#### Add the role ARN as a GitHub repository secret:

1. Go to the repository **Settings > Secrets and variables > Actions**
2. Create a new secret: `AWS_ROLE_ARN`
3. Value: `arn:aws:iam::728131696474:role/nf-core-github-actions`

### 4. Make the GHCR package public

After the first deploy pushes an image:

1. Go to the GitHub package at
   `https://github.com/orgs/nf-core/packages/container/nf-core-slackbot`
2. **Settings > Danger zone > Change visibility** to Public

This allows ECS to pull the image without needing registry credentials.

### 5. First deployment

Push to `main` to trigger the GitHub Actions workflow, or run CDK manually:

```bash
cd infra
pip install -r requirements.txt
cdk deploy --profile nf-core --require-approval never
```

## Manual CDK Operations

For debugging or one-off infrastructure changes:

```bash
cd infra
pip install -r requirements.txt

# Preview what CDK would change
cdk diff --profile nf-core

# Deploy
cdk deploy --profile nf-core

# View stack outputs (cluster name, service name, table name)
cdk deploy --profile nf-core --outputs-file outputs.json

# Destroy everything (DynamoDB table is retained by default)
cdk destroy --profile nf-core
```

## Forcing a Redeployment

If you need to restart the bot without code changes (e.g. after rotating
secrets):

```bash
SERVICE=$(aws ecs list-services --cluster nf-core-bot --region eu-west-1 --profile nf-core --query 'serviceArns[0]' --output text)
aws ecs update-service --cluster nf-core-bot --service "$SERVICE" --force-new-deployment --region eu-west-1 --profile nf-core
```

## Cost

- **Fargate** (0.25 vCPU, 512 MB, 24/7): ~$9-10/month
- **DynamoDB** (on-demand): negligible for expected volume
- **VPC**: $0 (public subnets only, no NAT gateway)
- **GHCR**: free for public packages
- **Total**: ~$10/month
